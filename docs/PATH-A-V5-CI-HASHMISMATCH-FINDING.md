# PATH-A v5 — CI HashMismatch finding

**BLUF:** Today's v5 install failed because of TWO compounding choices made by the SRE-Windows v5 review, BOTH of which deviated from what worked in Session 19 (2026-05-08):

1. Used the **WHQL-overlay-INTACT** 78,424-byte variant (`0d9a89d0…`) on the assumption that preserving Apple's WHQL Authenticode signature was important for reproducibility. The signature is invalid because we patched bytes (HashMismatch), but the cert table is still there.
2. Used **pnputil INF install** path instead of **PATCH-APPLE-SYS direct-copy**. pnputil triggers CI to validate the PE Authenticode signature; direct-copy does not.

**Yesterday's working setup (Session 19) used the OPPOSITE choices:**
1. **WHQL-overlay-STRIPPED + re-signed by M14** (66,288 B, MD5 `c881c041…`) — signtool stripped Apple's old WHQL cert table and wrote a new M14-signed Authenticode entry. The new entry's hash matches the patched bytes.
2. **PATCH-APPLE-SYS direct-copy** to `C:\Windows\System32\drivers\applewirelessmouse.sys` — kernel loads via existing service ImagePath, no pnputil PE-signature check.

**This narrowly invalidates the v5 pnputil INF install method with overlay-intact strategy, NOT the descriptor patch concept.** The descriptor patch itself is fine. The Session 19 install method remains viable.

Our v5 INF (`oem53.inf`) was rank `0xFF0000` — outranked Apple-style and inbox. **PnP selected our INF.** The bind succeeded in the registry sense; the kernel load failed at CI verification.

### Conclusion (this finding)
On **Windows 11 build 26200 with testsigning ON + M14 cert in TrustedPublisher**:
- A patched PE with corrupted embedded Microsoft WHQL Authenticode signature (HashMismatch)
- Plus a valid testsigned `.cat` that hashes the patched bytes
- → **CI rejects the load** with `STATUS_DEVICE_NOT_CONNECTED` (0xC000009D) downstream of the .sys load failure

This empirically answers PSN-0001 H-024 (which was "NOT YET TESTED"): **the gate-zero question is FAIL.** The cat fallback does not override the embedded-signature hash check on this OS build.

## What this DOES NOT invalidate

### PATH-A descriptor patch concept (still alive)
The 116-byte HID descriptor patch at file offset `0xA850` itself is fine. Session 19 (2026-05-08) PacketLogger confirmed the patched filter delivers `[90 04 28 ...]` = 40% buf[2] battery readout on v3 single-shot.

### PATCH-APPLE-SYS direct-copy route (still works)
The existing `PATCH-APPLE-SYS|<nonce>|<patched-sys-path>` queue route in `mm-task-runner.ps1`:
1. Stops the applewirelessmouse service
2. Copies the patched `.sys` directly into `C:\Windows\System32\drivers\applewirelessmouse.sys`
3. Restarts the device

**This bypasses pnputil's CI gate entirely.** The kernel loads the file via the existing service registration. Yesterday this loaded the patched binary cleanly and battery worked.

### Re-sign-on-the-.sys strategy (still works)
The `SIGN-FILE|<nonce>|<file>|<thumbprint>` queue route uses `signtool sign /sm /sha1 ...` directly on the .sys. signtool strips the prior cert table (Apple's WHQL) and writes a new M14 Authenticode entry whose hash matches the patched bytes. CI sees a valid signature on the patched bytes — load succeeds.

The BSOD #1 + #2 incidents under this path were caused by the **F2/FX runtime bug** (NULL+len=4 race in IOCTL 0x410003 completion callback when multi-TLC descriptor active), NOT by signature/load failures. Different problem domain.

## BSOD attribution correction (added 2026-05-09 post-test review)

**The BSOD #1 + #2 root cause is NOT the descriptor patch; it is aggressive HID I/O sweeping triggering the latent NULL-deref in Apple's IOCTL 0x410003 BRB completion path.** PRD-184 E_S1 static analysis explicitly identifies BRB cancellation paths as the firing site for the `applewirelessmouse+0x9f0e` NULL deref. Session 19 (2026-05-08) confirmed the patched filter works fine under normal use — battery, cursor, scroll all functional. The crashes followed RID-sweep probe scripts or lifecycle disruption events that hit the cancellation/race window.

**Operational rule:** do NOT run RID-sweep / brute-force HID Feature probing scripts against the live driver. Use single-shot reads only. Diagnostic captures should use ETW/tracelog (read-only), not active HID I/O against the device.

This means:
- PATH-A descriptor patch is **operationally safe** with the right hygiene
- The "F2/FX runtime bug" is real but **trigger-controlled**, not always-firing
- Earlier SRE-Windows review framing of "PATH-A architecturally dead" was wrong — it's "PATH-A safe with operational discipline; aggressive probing is the BSOD trigger, not the patch"

## PATH-C (SDP cache patch) ruled out for v3

Cross-session work on the keyboard PRD-185 successfully used a 4-byte SDP cache registry patch. **That approach does NOT translate to v3** due to firmware-protocol asymmetry:

- v3 raw descriptor declares exactly ONE Feature: RID 0x55, 64 bytes (touchpad gestures, vendor 0xFF02 TLC)
- col01 FeatLen=2 reported by `probe-all` is the phantom RID 0x47 Feature added by `applewirelessmouse.sys` at runtime in Mode B
- v3 firmware does NOT respond to ANY Feature GET_REPORT (probe-all returns err=87 for RID 0x47, confirming phantom-only declaration)
- Battery on v3 is ONLY readable via `HidD_GetInputReport(RID 0x90) byte[2]` in Mode A (per Session 19 PacketLogger)

Adding Feature declarations to the v3 SDP cache would just create more phantom decls like applewirelessmouse already does — they don't connect to wire responses. The only way to get v3 battery is via the kernel filter producing the COL02 vendor-battery TLC and supporting `HidD_GetInputReport(0x90)` Input Report flow. **PATH-A descriptor patch is therefore the only viable path for v3 battery.**
